# AI Chat Gap Analysis 与下一阶段 Backlog 设计

- 日期：2026-03-30
- 项目：ai-chat
- 目的：基于当前代码、`PROJECT_PLAN.md` 与现有设计/计划文档，识别尚未完成的任务与下一阶段推荐顺序

## 1. 背景

当前仓库的主闭环已经基本完成：

- monorepo / Turborepo / shared package 已落地
- auth、JWT、基础 RBAC 已落地
- chat 会话、消息查询、SSE 流式聊天已落地
- tool execution 已支持持久化与前端展示
- schedule 与 runs 的 API、页面与自动 tick 执行已落地

因此本阶段的重点不再是“搭主骨架”，而是识别三类差距：

1. 文档中明确承诺但当前代码尚未落地的缺项
2. 已经实现但仍明显偏 MVP、可用性不足的半成品项
3. 从 MVP 走向更完整系统时应进入下一阶段的增强项

## 2. 分析口径

本次差距分析采用以下口径：

- 以当前代码实现为准判断真实现状
- 结合 `PROJECT_PLAN.md` 判断原始目标是否完成
- 结合已有 design / plan 文档判断是否存在文档与代码偏差
- 输出结果同时覆盖“差距清单”和“推荐 backlog”

本次分析不把早期文档中的所有扩展设想都视为必须补齐的缺陷；优先关注当前项目边界内、且对后续开发判断有实际影响的差距。

## 3. 差距分类设计

建议将当前尚未完成或尚未收口的事项拆成三组，而不是混成一个列表。

### 3.1 A 组：明确缺项

这组指文档里已经清楚承诺，但当前代码中尚未落地的能力。

当前识别到的主要项：

- `settings` 页面仍未实现
- `audit log` 模块与持久化未实现
- `refresh token` 机制未实现
- 设计文档与代码现状存在未同步项

这组任务的特点是：是否完成相对容易判断，适合用来补“承诺账”。

### 3.2 B 组：明显半成品

这组指主能力已经存在，但产品可用性、完整性或信息表达还偏薄。

当前识别到的主要项：

- `schedules` 页面仍偏基础 CRUD 形态
- `runs` 页面已有列表和详情，但查看、筛选、跳转体验较薄
- chat / schedule / run 的串联关系可见度不够
- tool 体系已可用，但工具种类仍较少、与早期文档设想不完全一致

这组任务的特点是：系统已经能跑，但用户感知上仍有“做完了 70%，还没到顺手”的感觉。

### 3.3 C 组：生产化增强

这组指从 MVP 走向更完整系统时需要补的基础设施与部署能力。

当前识别到的主要项：

- 接口限流
- 统一 timeout / 执行边界
- 错误追踪与结构化日志
- API / worker 进程拆分
- 管理员全局视角扩展

这组任务不应混入当前的 gap-closing 阶段，否则会显著扩大范围。

## 4. 核心差距结论

### 4.1 明确未完成的原始承诺项

#### 4.1.1 `settings` 页面

`PROJECT_PLAN.md` 明确列出了 `/settings` 页面以及默认模型、温度、最大步数、默认 system prompt、tool 开关等设置项，但当前前端路由中尚无 `/settings`。

这是当前最明确、最直接的“文档已承诺但代码未实现”的缺项。

#### 4.1.2 `audit log`

原始方案和设计中将审计日志视为核心模块与核心表之一，但当前后端模块中尚无独立 `audit` 实现，也没有对应的查询/落库能力。

该项更偏后端治理能力，适合作为第二阶段能力补强。

#### 4.1.3 `refresh token`

原始方案中对鉴权的正式建议包含 `access token + refresh token`，但当前认证链路只返回 `accessToken`。

该项不影响当前 MVP 运行，但属于从“能用”走向“更正式”的明确缺口。

### 4.2 已做但仍明显偏薄的项

#### 4.2.1 schedules 页面

当前页面已支持创建、编辑、启停、删除，但仍偏向最小 CRUD：

- 类型输入与参数表达还可更清晰
- `nextRunAt` / `lastRunAt` 的信息传达仍可更直观
- 列表态与编辑态的切换体验还有提升空间

#### 4.2.2 runs 页面

当前页面已支持筛选、列表、详情，但还存在明显的 MVP 痕迹：

- 详情结构比较基础
- 与 chat session 的跳转关系还未形成顺滑闭环
- 空态、错误态、筛选体验仍可增强

#### 4.2.3 tool 体系

当前 tool 体系已经可用，并已超出最初单一 `get_current_time` 的阶段，但和早期计划中对工具种类的设想仍有差距。

同时，代码中已经出现 `manage_schedule` 以及 `INTERVAL` 支持，说明当前真实实现已经领先于部分旧文档。

### 4.3 文档与代码存在偏差

当前最明显的偏差包括：

- schedule 早期设计文档主要围绕 `CRON` 与 `ONE_TIME`，但代码已支持 `INTERVAL`
- tool 设计曾强调 `create_schedule_task` 等拆分式工具，但当前实现更倾向于用 `manage_schedule` 聚合处理
- 早期总体文档中的部分扩展模块仍停留在设想层，容易被误读为“已经存在”

因此，“文档同步”本身应被视为一个正式任务，而不是顺手备注。

## 5. 方案对比

### 方案 1：对齐文档优先

目标：先补齐文档承诺项。

包含：

- `/settings` 页面
- `audit log`
- `refresh token`
- 文档同步

优点：

- 与原始文档对齐度最高
- 哪些承诺已兑现更清楚

缺点：

- 对当前用户感知最强的问题提升不一定最大
- 后端改动面会更早扩大

### 方案 2：产品可用性优先

目标：先把现有 MVP 打磨成更顺手的版本。

包含：

- `/settings` 页面
- `schedules` 页面增强
- `runs` 页面增强
- chat / schedule / run 串联体验增强
- 文档同步

优点：

- 对现有系统收益最直接
- 延续当前代码边界扩展，风险更可控
- 可以先把最明显的缺口补到一个更顺手的状态

缺点：

- `audit log` 与 `refresh token` 会继续后置

### 方案 3：聊天协议统一优先

目标：优先把 chat 主链路升级到更完整的消息协议与前端消费模型，直接对齐 Vercel AI SDK 的聊天体验。

包含：

- chat 前后端协议统一到 AI SDK 兼容流式消息协议
- Web 侧改为 `@ai-sdk/react` + `useChat` 消费消息流
- assistant 文本、tool 调用、tool 结果融入同一 message parts
- 保留 session / chat message / tool execution 持久化
- 文档同步

优点：

- 直接解决当前 chat 最核心的体验缺口
- 不再长期维护当前自定义 SSE 事件协议与前端手写分发逻辑
- 为后续 tool 扩展、多轮对话与更完整 agent 体验打下统一基础

缺点：

- 变更面显著大于单纯 UI 打磨
- 会影响 API chat 输出、前端 chat 状态管理与测试基线

### 方案 4：生产化优先

目标：先补系统治理与部署能力。

包含：

- `refresh token`
- 限流
- timeout
- 错误追踪 / 结构化日志
- API / worker 拆分
- `audit log`

优点：

- 更接近正式部署形态
- 更利于后续扩展

缺点：

- 范围更重
- 对当前“功能完成度”的提升不如方案 2 和方案 3 直接

## 6. 推荐方案

推荐采用 **方案 3：聊天协议统一优先**。

理由：

1. 当前 chat 的主要问题已经不是“有没有聊天页”，而是消息流表达仍停留在自定义事件 + 独立 tool 列表阶段
2. 直接把 chat 主链路统一到 AI SDK 兼容消息协议，更符合你要的 ChatGPT 式多轮消息体验
3. 相比额外增加协议适配层，直接收敛主链路虽然改动更大，但长期维护成本更低
4. session、message、tool execution 的持久化基础已经存在，这一轮可以集中改 chat 交互链路，而不必同步重写数据库模型

## 7. 推荐 backlog

### 7.1 第一阶段：chat-protocol-upgrade

目标：先把 chat 主链路升级成更完整的消息协议与更自然的多轮消息体验。

包含：

1. 文档同步
2. chat 前后端协议统一到 AI SDK 兼容流式消息协议
3. Web chat 页面迁移到 `@ai-sdk/react` + `useChat`
4. assistant message 内嵌 tool 调用 / tool 结果 / 错误状态
5. 保持 session / chat message / tool execution 持久化兼容

#### 7.1.1 文档同步边界

当前真实状态已同步为：

- chat 前后端主链路已切换到 AI SDK 兼容流式消息协议
- Web chat 页面已迁移到 `@ai-sdk/react` + `useChat`
- tool 调用、结果、失败状态已内嵌到 assistant message
- 未引入独立协议适配层；数据库仍保持文本 `ChatMessage` + `ToolExecution` 持久化模型
- schedule 已支持 `INTERVAL`
- `runs` / `schedules` 协议本阶段未改动

只做与当前真实现状直接相关的同步：

- 标明已经完成的主闭环
- 更新 schedule 已支持 `INTERVAL`
- 更新 tool 现状
- 明确 chat 已从自定义 SSE 事件模型切换到 AI SDK 兼容消息协议
- 明确仍未完成的项

不重写所有历史文档，不把本轮工作变成全面文档整理。

#### 7.1.2 chat 协议升级边界

本阶段直接调整现有 chat 主链路，不新增长期存在的独立协议适配层。

优先包含：

- API 原生输出 AI SDK 兼容流式消息协议
- 前端改为消费统一的 message parts
- assistant 文本、tool 调用、tool 结果、tool 失败融入同一消息流
- 保留当前 session / chat message / tool execution 持久化

本阶段不做：

- 历史消息 100% 重建为完整 parts 结构
- `/runs` 实时流式详情页
- `/schedules` 页面协议同步改造
- 数据库 schema 的 parts 化重构

#### 7.1.3 Web chat 页面边界

优先增强：

- 连续多轮对话观感
- 更自然的 assistant streaming 体验
- tool 区块融入 assistant message
- 会话切换后的历史消息兼容展示

本阶段不做：

- 复杂消息编辑能力
- 多模态附件系统
- 自定义拖拽工作台式消息布局

### 7.2 第二阶段：gap-closing

目标：补齐除 chat 主链路外，用户最直观可感知的缺口。

包含：

1. `/settings` 页面
2. `/runs` 页面增强
3. `/schedules` 页面增强

#### 7.2.1 `/settings` 页面边界

只展示当前系统真正有承接能力的配置项，不做“后端尚未支持但先摆 UI”的伪配置。

优先候选包括：

- 默认模型
- 默认 system prompt
- 最大 steps
- tool 开关

如果温度等配置尚未有稳定后端承接，则不纳入第一版 `/settings`。

#### 7.2.2 `/runs` 页面边界

优先增强：

- 更清晰的详情信息结构
- `chatSessionId` 关联跳转
- 更合理的筛选 / 空态 / 错误态

本阶段不做：

- run 实时流式详情页
- 复杂管理员面板
- 批量操作

#### 7.2.3 `/schedules` 页面边界

优先增强：

- schedule 类型与参数的更清晰表达
- `nextRunAt` / `lastRunAt` 的更直观展示
- 更顺的编辑态与列表态切换

本阶段不做：

- 手动立即执行
- 批量启停
- 模板化任务系统
- 工作流编排

### 7.3 第三阶段：后端治理增强

目标：补后端可信度与正式化能力。

包含：

1. `refresh token`
2. `audit log`
3. 关键接口限流与 timeout

这三项都属于系统治理层改动，适合在同一阶段统一处理。

### 7.4 第四阶段：扩展与部署形态

目标：补充扩展能力与生产化部署边界。

包含：

1. API / worker 进程拆分
2. 更实用的 tool 扩展
3. 管理员全局 schedules / runs 视角
4. 错误追踪与结构化日志

## 8. 最终结论

当前项目的主 MVP 已经基本完成。真正尚未完成的核心，不再只是传统意义上的页面缺项，而是：

- 文档承诺项中的明确缺口（尤其是 `/settings`、`audit log`、`refresh token`）
- chat 主链路仍停留在自定义 SSE 事件分发与分离式 tool 展示阶段
- `schedules` / `runs` / tool 这些已经可用但仍偏薄的模块
- 从 MVP 走向更完整系统所需的治理与部署增强项

因此，下一轮工作的最佳切入点不是继续补零散页面，而是先完成一轮 **chat-protocol-upgrade 阶段**：

> 先做“文档同步 + chat 前后端协议统一 + ChatGPT 式消息体验升级”，再进入“settings / runs / schedules 增强”，最后进入“auth / 审计 / 稳定性增强”阶段。

这样既能先解决当前最核心的聊天体验缺口，又能保持后续 backlog 的阶段边界清晰，符合当前项目的 KISS 原则。
