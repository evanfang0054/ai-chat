# Chat MVP 设计方案

- 日期：2026-03-26
- 项目：ai-chat
- 阶段：任务 2 / Chat MVP

## 1. 目标

在已完成账号体系、JWT 鉴权、基础 RBAC 与 Web 登录闭环的基础上，交付一个可真实使用的聊天 MVP。

本阶段目标是完成：
- 多会话聊天
- 会话与消息持久化
- DeepSeek 真实模型接入
- SSE 流式返回
- Web 端类 ChatGPT 聊天页
- 按用户隔离聊天数据

本阶段不实现 Tool、Schedule、统一运行记录持久化、附件上传、消息编辑、停止生成等扩展能力。

## 2. 设计原则

- KISS：优先交付最小可用聊天闭环
- 真实链路优先：本阶段直接接入 DeepSeek，而不是 stub
- 边界先立住：虽然只做 Chat MVP，但代码结构先对齐后续 `chat -> run -> llm -> tool` 演进方向
- 不过度实现：本阶段只建立最小 runtime 边界，不提前做完整 agent runtime
- 用户隔离优先：所有会话与消息必须绑定用户并隔离访问

## 3. 范围与非目标

### 3.1 本阶段范围
- 新增聊天后端模块
- 新增最小 `run` 模块，作为一次聊天执行的统一入口
- 新增 `llm` 模块，封装 DeepSeek 调用
- 新增聊天数据模型 `ChatSession` / `ChatMessage`
- 新增聊天 Web 页面 `/chat`
- 新增会话列表、消息区、输入框、空态、流式渲染
- 消息内容支持 Markdown 基础展示

### 3.2 明确不做
- Tool 实际执行
- Schedule 任务触发聊天
- `Run` 数据表
- 附件上传
- 消息编辑/删除
- 停止生成
- 多模型切换 UI
- 第三方登录

## 4. 总体架构

本阶段采用以下执行链：

- 前端：`/chat` 页面负责展示会话列表、消息流和输入框
- 后端入口：`chat` 模块负责会话与消息业务编排
- 执行中层：`run` 模块负责组织一次聊天执行上下文
- 模型调用：`llm` 模块负责调用 DeepSeek 并输出流式文本

本阶段的核心链路为：

`chat -> run -> llm`

未来需要接入 Tool 或 Schedule 时，再扩展为：

`chat / schedule -> run -> llm / tool`

本阶段不为未来能力引入数据库级 runtime 抽象，只先把代码边界建立起来。

## 5. 后端模块设计

### 5.1 chat 模块
职责：
- 提供会话列表接口
- 提供消息历史接口
- 提供发送消息接口
- 提供 SSE 流式响应接口
- 创建会话与生成标题
- 保存用户消息与 assistant 消息
- 做当前用户的数据访问校验

`chat` 模块不直接调用 DeepSeek，而是把一次聊天执行交给 `run` 模块。

### 5.2 run 模块
职责：
- 接收当前会话、历史消息、当前输入
- 组织发送给模型的上下文
- 调用 `llm` 模块
- 接收增量 token/文本片段
- 把流式输出通过回调或事件持续回传给 `chat`

本阶段的 `run` 模块仅作为代码边界，不落地数据库表，不承担队列、调度、审计或重试职责。

### 5.3 llm 模块
职责：
- 读取服务端环境变量中的 DeepSeek 配置
- 封装 DeepSeek 聊天接口调用
- 提供流式文本生成能力
- 统一输出增量文本事件给 `run`

本阶段只接一个真实模型提供方，不做多提供方抽象层。

### 5.4 tool 模块
本阶段只保留未来接口边界的认知，不创建真实 tool 执行链。任务 2 中不实现 tool 注册、调用或结果回写。

## 6. 数据模型设计

### 6.1 ChatSession
建议字段：
- `id`
- `userId`
- `title`
- `model`
- `createdAt`
- `updatedAt`

约束：
- 每个会话必须归属于一个 `User`
- `title` 由首条用户消息截断生成
- `updatedAt` 用于会话列表按最近活跃排序

### 6.2 ChatMessage
建议字段：
- `id`
- `sessionId`
- `role`
- `content`
- `createdAt`

约束：
- 每条消息必须归属于一个 `ChatSession`
- `role` 本阶段建议支持 `USER`、`ASSISTANT`，可保留 `SYSTEM` 枚举位但不要求在 UI 中使用
- assistant 的最终消息在流完成后落库

### 6.3 本阶段不新增 Run 表
虽然代码结构中会加入 `run` 模块，但本阶段不增加 `Run` / `ChatRun` 数据表。

原因：
- 任务 2 只需要先把执行链边界建起来
- 任务 3/4 才需要决定统一运行记录是否值得持久化
- 现在引入 runtime 表容易把 MVP 做重

## 7. 后端接口与数据流

### 7.1 会话列表
- 获取当前用户会话列表
- 仅返回当前登录用户自己的会话
- 默认按 `updatedAt desc` 排序

### 7.2 会话消息历史
- 获取指定会话的历史消息
- 若会话不属于当前用户，返回 404 或拒绝访问

### 7.3 发送消息
发送消息时的后端行为：
1. 校验当前用户身份
2. 如果当前没有会话，则先创建 `ChatSession`
3. 用首条用户消息截断生成会话标题
4. 保存用户消息
5. 查询该会话历史消息
6. 调用 `run` 模块
7. `run` 调用 `llm` 模块访问 DeepSeek
8. 流式输出通过 SSE 回给前端
9. 流结束后保存 assistant 最终消息
10. 更新会话 `updatedAt`

### 7.4 SSE 流式返回
SSE 负责把 assistant 增量文本持续返回到前端。

本阶段只要求支持：
- 文本增量事件
- 结束事件
- 错误事件

不要求支持：
- tool call 事件
- reasoning 事件
- token usage 细分事件

## 8. DeepSeek 集成策略

### 8.1 配置来源
DeepSeek 凭据与配置只允许从服务端环境变量读取，例如：
- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`（可选）
- `DEEPSEEK_MODEL`（可选，未配置时使用默认模型）

### 8.2 安全约束
- API key 不写入仓库
- API key 不出现在前端代码中
- API key 不写入设计文档中的真实值
- 本地开发通过 `.env` 或 shell 环境注入

### 8.3 模型调用范围
本阶段只支持：
- 文本输入
- 文本流式输出
- 基于历史消息上下文的多轮聊天

不做：
- function/tool calling
- 文件理解
- 图像输入
- 多模型路由

## 9. 前端页面设计

### 9.1 页面路由
新增 `/chat` 页面，并纳入现有 `ProtectedRoute`。

建议登录后默认落到 `/chat`，保留 `/admin` 管理页面不变。

### 9.2 页面布局
采用类 ChatGPT 布局：

左侧 Sidebar：
- `New Chat` 按钮
- 当前用户的会话列表
- 当前会话高亮

右侧 Main Panel：
- 顶部标题区（会话标题 / 模型信息）
- 中部消息流
- 底部输入框与发送按钮

### 9.3 交互规则
#### 首次进入 `/chat`
- 拉取当前用户会话列表
- 若存在会话，自动进入最近更新的会话并加载历史消息
- 若不存在会话，展示空态欢迎区

#### 点击 `New Chat`
- 清空当前右侧聊天内容
- 进入“待开始新会话”状态
- 只有在用户发送第一条消息时才真正创建会话

#### 发送消息
- 前端先插入本地 `USER` 消息以提升即时反馈
- 若还没有会话，则走“创建会话 + 首条消息 + 流式回复”链路
- 建立 SSE 连接接收 assistant 增量文本
- 前端用一条临时 assistant 消息持续拼接展示
- 流结束后用服务端最终消息完成同步

#### 切换会话
- 请求目标会话的历史消息
- 替换当前消息区内容
- 若上一个会话仍有流式返回，前端应中断或忽略旧流结果

## 11. 错误处理策略

### 11.1 后端
- DeepSeek 调用失败：
  - 发送 SSE 错误事件
  - 不保存失败的 assistant 消息正文
  - 已保存用户消息保留
- 会话不存在或越权：
  - 返回 404 或拒绝访问
- 空消息或非法输入：
  - 返回 400
- API key 缺失：
  - 服务端明确报错
  - 不向前端暴露敏感配置细节

### 11.2 前端
- 会话列表加载失败：显示错误态与重试入口
- 历史消息加载失败：显示局部错误提示
- 流式发送失败：保留输入内容并提示失败
- 正在发送时：禁止重复提交同一输入
- 切换会话时：中断或忽略旧流

## 12. 测试策略

### 12.1 API
至少覆盖：
- 当前用户可创建会话
- 会话列表只返回当前用户自己的数据
- 会话消息历史只返回当前用户自己的数据
- 发送消息时可正确保存用户消息
- `run` 与 `llm` 交互可通过 mock `llm` service 做模块测试
- SSE 接口可返回流、结束事件、错误事件
- 越权访问他人会话会失败

### 12.2 Web
至少覆盖：
- 未登录访问 `/chat` 会跳转 `/login`
- 已登录用户可进入聊天页空态
- 会话列表切换会刷新消息区
- 发送消息时本地用户消息立即可见
- assistant 流式消息会逐步更新
- 错误态会显示

### 12.3 联调
至少验证：
1. 登录
2. 进入 `/chat`
3. 新建会话并发送首条消息
4. 页面接收并展示流式回复
5. 刷新后历史消息仍可恢复
6. 普通用户无法读取其他用户会话

## 13. 验收标准

本阶段完成时应满足：
- 用户登录后可进入 `/chat`
- 用户可以创建多个会话并切换
- 会话标题由首条用户消息截断生成
- 用户消息与 assistant 消息会持久化到数据库
- 页面刷新后历史聊天可恢复
- DeepSeek 可真实返回流式文本
- 会话与消息严格按用户隔离
- 前端页面采用类 ChatGPT 布局
- 敏感 key 不进入 git，仅通过本地环境变量配置

## 14. 风险与控制

### 风险 1：过早把 runtime 做重
控制：
- `run` 只作为代码边界
- 不新增 runtime 数据表
- 不引入调度、重试、状态机

### 风险 2：把任务 2 和任务 3 混在一起
控制：
- 本阶段不做真实 tool
- 不做 tool 状态事件
- 不做 tool 执行持久化

### 风险 3：流式链路前后端状态不一致
控制：
- 前端仅将流式 assistant 消息视为临时状态
- 以后端最终持久化结果为准

## 15. 结论

本设计将 Chat MVP 定义为：

**带最小 runtime 边界的真实多会话流式聊天闭环。**

它满足以下平衡：
- 有真实价值：直接接 DeepSeek，是真聊天，不是假数据
- 有正确方向：先立 `chat -> run -> llm` 边界
- 不过度设计：不把 Tool、Schedule、Run 持久化提前做出来

这样可以在当前仓库已完成鉴权闭环的基础上，最稳妥地进入下一阶段，并为后续 Tool MVP / Schedule MVP 留出自然扩展路径。
